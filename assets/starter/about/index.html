About this starter

A working app that ships with the DSGo Apps plugin so you can install something real with one click and learn by reading the source.

Where the bundle lives

After install, the bundle is extracted to wp-content/uploads/dsgo-apps/dsgo-starter/. You can edit the files there directly, or replace the whole bundle by uploading a new .zip on the Apps page.

The manifest, annotated

Every DSGo App is described by a single dsgo-app.json at the root of its bundle. Here are the fields this starter declares:

id, name, version, description

id becomes the URL segment (/apps/dsgo-starter/) and the post slug. name shows in the admin Apps list. version is SemVer 2.0. description is metadata only.

entry + routes

Inline-mode apps declare every page they serve in routes. Each entry maps a URL path to a static HTML file in the bundle. The first route must be /. Paths support a single :param placeholder for dataset-backed dynamic routes (not used here).

isolation: "inline"

This starter uses inline isolation: each route is served by WordPress with a fresh per-request CSP nonce stamped onto every <script>. The plugin auto-injects window.dsgo so apps don’t bundle a client. The alternative, "iframe", gives you an opaque-origin sandbox and is required for abilities.publishes.

permissions.read

An allowlist of bridge surfaces the app may use. The bridge enforces these server-side; calling something not in the list returns permission_denied. This starter declares all 7 (site_info, posts, pages, user, abilities, ai, email) so every demo works.

permissions.write exists in the schema but must be empty in v1. Writes (create/update/delete) come in v2.

abilities.consumes

Required when abilities is in permissions.read. A list of glob patterns (e.g. "yoast-seo/*", "wp/*") saying which abilities this app may invoke. The bridge filters abilities.list() against this allowlist.

email.recipients

Required when email is in permissions.read. Either "admin" (the site admin email) or "current_user" (the logged-in visitor). The app never sees the actual address — the bridge resolves it.

runtime.csp

The four required CSP directives (script_src, style_src, img_src, connect_src) plus optional font_src. Each must include "self". To call an external API, add the origin to connect_src. The plugin builds the actual Content-Security-Policy header from this block per request.

Want a real dev workflow?

From your terminal:

npx designsetgo apps init my-app
cd my-app
npx designsetgo apps login
npx designsetgo apps deploy --build

That scaffolds an Astro project with hot reload, deploys via an Application Password, and is the recommended path once you outgrow this hand-crafted starter.
